# 尾部局部优化器算法升级设计

## 背景与目标

当前项目的单 SKU 装柜算法已经能稳定生成 2D/3D 可视化坐标，并支持柜型、车厢间隙、角件避让、批量导入和多 SKU 分配。但它本质上是“规则化码垛候选枚举”，不是完整的几何最大装载搜索。

用户提供的 `码垛算法差异.xlsx` 中有三条对比数据，说明现有算法在柜门侧尾部补位上偏保守：

| 纸箱尺寸 | 柜型 | 当前结果 | 目标结果 | 差值 |
| --- | --- | ---: | ---: | ---: |
| `488 × 380 × 291 mm` | `40HQ` | `1,340` | `1,349` | `+9` |
| `509 × 418 × 338 mm` | `40HQ` | `875` | `889` | `+14` |
| `536 × 436 × 330 mm` | `40HQ` | `919` | `927` | `+8` |

本次迭代目标是把单 SKU 最大装载算法升级到可以达到上述目标结果，同时保留现有坐标输出、角件避让、批量导入和 2D/3D 渲染能力。

## 为什么这样做

现有算法的主体能力可复用，不需要推倒重写：

- `calculatePacking(...)`、`calculatePackingTotalBoxes(...)`、`generateBoxPositions(...)` 已经是页面和批量导入依赖的稳定入口。
- `src/core/packing/candidates.ts` 已经承担候选排布生成，适合继续扩展候选空间。
- 角件避让、碰撞检测、装载顺序和渲染坐标已经有测试覆盖，推倒重写会增加回归风险。

真正的差异集中在“尾部局部重排”：对方算法允许减少主排布某些巷道的纵深，在柜门侧释放出一个较大的尾部区域，再在该区域内做局部混排。本项目当前只做一次门端剩余空间补位，不会对补位区域进行更完整的局部搜索。

## 做什么

### 1. 保留外部 API

以下公开入口保持兼容：

```ts
calculatePacking(container, carton, options?)
calculatePackingTotalBoxes(container, carton, options?)
calculateMultiSkuPacking(container, skus, options?)
generateBoxPositions(result, visibleCount?)
```

页面、批量导入和渲染器不应感知新优化器的内部细节。算法返回的 `orderedPositions`、`layerPositions`、`pattern`、`layers`、`totalBoxes` 等结构继续作为事实来源。

### 2. 新增尾部局部优化器

在规则化候选基础上新增 `tail local optimizer`，建议作为独立模块放在 `src/core/packing/` 下，例如：

```text
src/core/packing/tailOptimizer.ts
src/core/packing/collision.ts      可选，若拆分现有碰撞/角件逻辑
```

优化器输入：

- `container`
- `carton`
- 允许朝向
- 一个基础候选排布或基础脚位集合
- 角件规格
- 优化参数，例如最多回退排数、最大搜索节点数

优化器输出：

- 优化后的单层脚位集合
- 新增或替换的尾部脚位
- 可解释的 `pattern` 元数据，例如 `tail-optimized`
- 诊断信息，例如回退了哪些巷道、补位数量、搜索节点数

### 3. 局部优化策略

优化器不做全局 3D bin packing。它只在单层平面上做小范围局部搜索，再按高度复制并经过角件避让校验。

建议流程：

1. 从现有 `width-lanes` 高分候选开始。
2. 对可回退的巷道尝试减少纵深，初始范围为 `1` 到 `3` 排；必要时按测试数据扩展到 `4` 排。
3. 用回退后释放的柜门侧区域生成候选矩形空间。
4. 在这些尾部空间内枚举允许朝向的箱体脚位。
5. 通过不重叠、不过柜、角件安全、门端边界检查筛选。
6. 使用小规模回溯或最大独立集选择最多可放脚位。
7. 把优化脚位合并回基础脚位，交给现有评估逻辑计算总箱数。

搜索必须有上限，避免批量导入时变慢：

- 单个候选最大搜索节点数固定上限。
- 只对接近当前最佳结果的候选启用局部优化。
- 只优化单 SKU 水平朝向场景；侧放朝向可以复用同一框架，但不作为本次验收重点。

### 4. 角件与三维校验

尾部优化器只负责生成候选脚位，最终结果必须继续通过统一评估：

- 箱体不能超出有效柜体空间。
- 同一层脚位不能重叠。
- 生成 3D 坐标后不能相互重叠。
- 顶部内端角件区域必须避让。
- 角件可通过现有“安全挪位”逻辑处理，但不能因为挪位造成新重叠。

`40HQ` 角件仍使用当前默认值：

```text
110 × 110 × 80 mm
```

### 5. 结果解释与可视化

优化后的结果仍然要能被 2D/3D 表达：

- `layerPositions` 必须包含尾部补位脚位。
- `orderedPositions` 必须包含完整装载顺序。
- 每个位置保留 `orientationId`、`orientationLabel` 或等价信息。
- 尾部补位位置可以标记 `source: "tail-optimized"`，方便策略说明和调试。

装载顺序保持当前约定：从柜内端向柜门端推进，同一深度内按层和宽向顺序生成。尾部补位不应破坏进度条和 3D 动画。

## 不做什么

本次迭代不做以下事项：

- 不推倒重写整个 `src/core/packing/` 算法。
- 不引入通用 3D bin packing 求解器。
- 不改变 20GP、40GP、40HQ 的柜型尺寸定义。
- 不改变默认朝向规则；默认仍只启用 `长×宽×高`、`宽×长×高` 两种水平旋转。
- 不重新设计 UI。
- 不把多 SKU 异尺寸混装升级为全局最优；多 SKU 先继承单 SKU 分区内核的能力。

## 验收标准

### 必须达到的新基准

新增或更新单元测试，确认以下结果：

| 纸箱尺寸 | 柜型 | 期望最大装载量 |
| --- | --- | ---: |
| `488 × 380 × 291 mm` | `40HQ` | `1,349` |
| `509 × 418 × 338 mm` | `40HQ` | `889` |
| `536 × 436 × 330 mm` | `40HQ` | `927` |

### 必须保持的既有基准

现有业务基准不能无意回退。若结果因“最大装载量优先”策略发生提升，需要在测试名称和预期里明确说明这是目标变更，而不是偶然变化。

重点关注：

- `488 × 360 × 291 mm`，`40HQ`，当前基准 `1,403`。
- 批量导入能返回新最大装载量和差值。
- `generateBoxPositions(result, result.totalBoxes)` 返回数量等于 `totalBoxes`。
- 所有生成箱体不与角件碰撞。
- 所有生成箱体在 3D 空间内不互相重叠。

### 验证命令

算法完成后优先运行：

```bash
npm run test:unit
npm run test:e2e
npm run build
```

如果环境限制导致某项验证无法执行，需要在最终说明中列出原因和已完成的替代检查。

## 风险与处理

### 风险一：局部搜索影响批量导入性能

处理方式：

- 对候选数量和搜索节点设置硬上限。
- 先用 `calculatePackingTotalBoxes` 走轻量评估。
- 对重复尺寸继续使用现有缓存。

### 风险二：补位数量正确但坐标不可视化

处理方式：

- 不接受“只算数量”的优化结果。
- 每个优化结果必须生成真实 `BoxPosition`。
- 测试同时检查数量、坐标边界、碰撞和角件。

### 风险三：追求最大量导致人工装柜不易执行

处理方式：

- 本次先按用户确认的目标，将单 SKU 最大装载升级为更高结果。
- 若后续业务需要，可再增加 `conservative` / `maximum` 模式切换；本次不把 UI 模式切换纳入范围。

### 风险四：`src/core/packing/index.ts` 继续膨胀

处理方式：

- 新优化逻辑放入独立模块。
- 只在 `index.ts` 中接入候选评估流程。
- 如需复用碰撞、角件、安全校验函数，可先做最小拆分，不做无关重构。

## 实施顺序建议

1. 为三条目标数据添加失败测试，先锁定新期望。
2. 为尾部优化器建立独立测试，验证单层脚位可达到目标补位数量。
3. 接入 `calculatePacking` 的候选评估流程。
4. 更新 `calculatePackingTotalBoxes`，保证批量导入结果一致。
5. 补充 3D 坐标碰撞和角件避让回归测试。
6. 运行单元测试、端到端测试和构建。

## 成功定义

本次迭代完成后，项目仍然保持当前页面和批量导入使用方式，但单 SKU 最大装载算法能达到用户提供的更高对比结果。结果不是只在数字上追平，而是有完整可视化坐标、装载顺序和角件安全校验支撑。
